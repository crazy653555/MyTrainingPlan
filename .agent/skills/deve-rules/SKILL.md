# 開發流程規範 (Development Rules)

這份文件定義了本專案的標準開發作業流程 (SOP)。當我（AI 助手）在處理使用者的開發需求時，必須嚴格遵守以下六個階段。在未獲得使用者明確同意前，絕對不可擅自跨越「停止點」進入下一個階段。

---

## 階段 1：需求討論與確認 (Requirement Discussion & Confirmation)
1. **分析需求**：釐清使用者的需求目的，盤點可能需要更動的檔案結構與技術架構。
2. **提出計畫**：向使用者提出具體的實作計畫、修改範圍與作法建議。
3. **重大功能評估**：如果分析後認為功能異動太大（例如：架構重構、新增大型模組），主動提議先建立新的 Git feature 分支。
4. **🛑 停止點**：等待使用者回覆並確認作法與分支策略是否正確。

## 階段 2：請求執行權限 (Permission to Execute)
1. **確認執行**：當作法確認無誤後。
2. **🛑 停止點**：明確詢問使用者：「請問我們現在可以開始執行程式碼的修改了嗎？」等待使用者給予明確同意（如：「開始」、「OK」）後再動作。若為重大異動，經同意後應先切換或建立分支再開始開發。

## 階段 3：執行開發與驗證 (Execution & Validation)
1. **實作程式碼**：依照確認的計畫進行程式碼開發。
2. **編譯與除錯**：開發告一段落後，主動確保程式碼沒有基礎層級的語法錯誤，並可視情況請使用者協助進行專案編譯或本地測試，確保不影響現有功能。

## 階段 4：同步更新文件 (Documentation Update)
程式碼開發完成並確保無誤後，**必須第一時間**同步更新專案的說明文件，確保程式碼與文件一致：
1. **SRS (軟體需求規格書)**：更新需求變更與功能描述。
2. **SDD (軟體設計文件)**：更新架構圖、API 規格、資料庫結構等設計細節。
3. **README.md**：若有新增依賴套件、啟動指令變更或核心功能說明，需同步更新。
*(註：請根據專案實際資料夾結構尋找並更新上述文件。)*

## 階段 5：成品與文件確認 (Change Confirmation)
1. **總結回報**：將修改的程式碼重點、更動的檔案清單，以及更新後的文件內容/連結，條理分明地總結回報給使用者。
2. **🛑 停止點**：明確詢問使用者：「以上程式碼與文件更新是否確認無誤？」等待使用者審閱並確認沒有遺漏。

## 階段 6：版本控管提交 (Git Commit & Push)
1. **確認版控**：收到使用者的無誤確認後。
2. **🛑 停止點**：詢問使用者：「任務已完成，請問是否需要為您執行 Git 提交 (commit) 操作？若在此之前有建立分支，是否需要一併詢問合併回主分支？」
3. **執行提交**：若使用者同意，請將變更提交。
   * **Commit Message 規範**：一律使用**繁體中文**做為標準語言。
   * **格式規範**：遵循 Conventional Commits 格式（例如：`feat: 增加登入功能`, `fix: 修復影片無法播放問題`, `docs: 更新 SDD 文件`）。
